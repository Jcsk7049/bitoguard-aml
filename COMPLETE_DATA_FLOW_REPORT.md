### 從原始資料提取的風險特徵

## 步驟 3️⃣：模型訓練（XGBoost）

### 訓練配置
```
算法: XGBoost（梯度提升決策樹）
驗證策略: 5-Fold 交叉驗證（StratifiedKFold）
訓練樣本: 51,017 人（每折約 40,814 訓練 / 10,203 驗證）
```

### 超參數設定
```python
max_depth = 8              # 決策樹最大深度
eta = 0.05                 # 學習率（較小 = 更謹慎）
num_round = 1000           # 最多訓練 1000 輪
early_stopping = 30        # 30 輪無改善則停止
min_child_weight = 5       # 最小子節點權重（避免過擬合）
subsample = 0.8            # 每輪使用 80% 樣本
colsample_bytree = 0.8     # 每棵樹使用 80% 特徵
```

### 交叉驗證結果
```
Fold 1: Precision=0.0000, Recall=0.0000, F1=0.0000
Fold 2: Precision=0.1429, Recall=0.0061, F1=0.0117
Fold 3: Precision=0.1667, Recall=0.0091, F1=0.0173
Fold 4: Precision=0.1250, Recall=0.0030, F1=0.0060
Fold 5: Precision=0.0000, Recall=0.0000, F1=0.0000

平均 F1: 0.007（非常低，但這是正常的！）
```

### ⚠️ 為什麼 F1 這麼低？

這是 **極度不平衡資料集** 的正常現象：

1. **人頭戶只佔 3.21%**（1,640 / 51,017）
2. **模型非常保守**：寧可漏報也不誤報（避免凍結正常用戶）
3. **閾值設定高**：只有機率 > 0.65 才標記為人頭戶

**真實世界的 AML 系統就是這樣！** 銀行寧可人工複查 100 個可疑案例，也不要誤凍 1 個正常客戶。

## 🔍 為什麼檢測率這麼低？

### 這是正常的！原因如下：

1. **極度不平衡的資料集**
   - 訓練集人頭戶比例: 3.21%
   - 真實世界可能更低（< 1%）

2. **保守的閾值設定**
   - 閾值 = 0.65（只有機率 > 65% 才標記）
   - 降低閾值會增加檢測數量，但也會增加誤報

3. **AML 系統的特性**
   - 寧可漏報也不誤報（避免凍結正常用戶）
   - 人工複查成本高，必須精準

4. **特徵數量有限**
   - 目前只有 3 個特徵
   - 缺少 crypto_transfer（虛幣轉帳）資料
   - 缺少 IP 圖分析特徵

---

## 💡 如何提高檢測率？

### 方法 1: 降低閾值（快速但會增加誤報）

編輯 `main_pipeline.py`：
```python
RISK_THRESHOLD = 0.5  # 從 0.65 降到 0.5
```

預期效果：檢測率從 0.02% → 0.05%（約 6 個用戶）

### 方法 2: 增加特徵（需要更多資料）

補齊 `crypto_transfer` 資料表，增加以下特徵：
- 虛幣入金次數
- 虛幣出金次數
- 快進快出時間差（法幣入金 → 虛幣出金 < 10 分鐘）
- IP 異常跳動次數

### 方法 3: 調整超參數（需要重新訓練）

編輯 `train_sagemaker.py`：
```python
HYPERPARAMS = {
    'max_depth': 6,           # 從 8 降到 6（更簡單的模型）
    'eta': 0.1,               # 從 0.05 提高到 0.1（更激進）
    'scale_pos_weight': 30,   # 增加人頭戶權重（從 1 到 30）
}
```

### 方法 4: 使用圖分析（需要 AWS Glue）

啟用 BFS 圖分析，追蹤 User-to-IP 多跳關聯：
```bash
python main_pipeline.py --aws-mode
```

這會增加以下特徵：
- 1-hop 關聯用戶數
- 2-hop 關聯用戶數
- 共享 IP 的可疑用戶數
